### ✅ 必须采纳的建议 (High Priority)

这几条直接关系到你论文的**逻辑自洽性**和**未来的 3D 扩展**。

#### 1. 风廓线公式的数值安全 (Fix the Math Bug)
*   **Copilot 指出**：`if z <= z0` 时硬编码 `0.1` 很滑稽，且 $\ln(z/z_0)$ 可能出负数。
*   **为何采纳**：这是数学硬伤。如果不改，审稿人一眼就能看出你的公式在贴地时是不连续的。
*   **行动**：使用它推荐的 `max(z, 1.5*z0)` 技巧。这能保证对数函数永远合法且平滑。
    *   **代码修改** (`estimator.py`)：
        ```python
        # 原来：if z <= z0: factor = 0.1
        # 改为：
        z_safe = max(z, 1.5 * z0) 
        factor = math.log(z_safe / z0) / math.log(h_ref / z0)
        ```

#### 2. 功率公式缺少重力项 (Gravity is Missing)
*   **Copilot 指出**：现在的功率公式只算了空气阻力，没算爬升时的重力做功 $mg \cdot v_z$。
*   **为何采纳**：这是 3D 规划的核心！如果不加这一项，无人机就会觉得“垂直爬升”和“水平飞行”能耗一样，那它肯定会无脑翻山，而不是绕路。
*   **行动**：虽然现在还是 2.5D，但如果你打算做 3D，这个公式必须预留好。
    *   **代码预埋** (`physics.py`)：
        ```python
        def calculate_power(self, v_ground, v_wind, climb_rate=0.0):
            # ... 前面算 v_air ...
            power_drag = ...
            power_gravity = self.config.drone_mass * 9.8 * climb_rate
            return power_drag + max(0, power_gravity) # 假设下降不回收能量
        ```

#### 3. 启发函数对垂直权重的忽视 (Heuristic Z-weight)
*   **Copilot 指出**：垂直飞 1米 比 水平飞 1米 累多了，启发函数 $h$ 必须体现这一点。
*   **为何采纳**：这是让 3D A* 跑得动的关键。如果不加权重，A* 会在垂直方向上浪费大量搜索步数。
*   **行动**：在 heuristic 里给 $z$ 轴加权。
    *   **代码预埋**：`dist = sqrt(dx^2 + dy^2 + (5 * dz)^2)`

### ⚠️ 可以采纳，但没必要现在做的 (Medium Priority)

这些属于“锦上添花”，做了能加分，不做也不扣分。

#### 1. 粗糙度映射的光滑过渡 (Smooth Transition)
*   **Copilot 建议**：用 Sigmoid 函数把森林和雪地的界限模糊化。
*   **评价**：**学术上正确，工程上没必要。**
    *   真实世界里，森林线（Tree line）往往就是很分明的。海拔一过，树就没了。用硬阈值（Hard Threshold）反而更符合某些高山特征。
    *   **结论**：保持现在的 `if-else` 或者 `np.where` 就行，解释起来也简单。

#### 2. 成本函数的积分形式 (Integral Cost)
*   **Copilot 建议**：用积分算 Cost，而不是简单的 `P * t`。
*   **评价**：**过度设计。**
    *   A* 本质上就是离散的。我们在网格间移动是直线的，风场假设也是线性的。用简单的乘法足够了。写成积分形式是为了发论文时好看，代码里没必要真去算积分。

### ❌ 不需要采纳的建议 (Over-engineering)

#### 1. 垂直气流分量 $W_{anabatic}$
*   **Copilot 建议**：风场应该是 $(u, v, w)$ 三维的，要算垂直风。
*   **评价**：**太难了。**
    *   垂直风场（Updraft/Downdraft）的建模非常复杂，如果不解 CFD，很难算准。
    *   对于大创来说，只考虑水平风随高度变化（风廓线）已经足够惊艳了。强行加垂直风容易画虎不成反类犬。
